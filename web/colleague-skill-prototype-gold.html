这份仓库在解决什么问题

人走了，聊天记录、文档、习惯还在四面八方。这里的思路是：把你能合法拿到的材料收成一套文件，让 AI 助手尽量按他以前的说话方式 + 工作套路来配合你——不是再训练一个小模型，而是一叠结构化的说明与脚本。

What this repo is for

When someone leaves, their voice and habits scatter across chats and docs. This repo packages what you can legitimately gather into files + prompts + scripts so an assistant can mirror their tone + work patterns — not fine-tune a model from scratch.

装好以后怎么用（与组件路径、模拟器对照）
1 · 技能进到宿主里按 README 把仓库放到宿主认的目录（如 .claude/skills/create-colleague）。没有常驻后台服务；只有你发出斜杠命令或同类请求时，才会跑里面的脚本与提示词。

GitHub 上为什么容易「星多」或看起来吵？

  • 需求：同事调岗/离职后，知识散落在聊天记录和文档里——这类「交接焦虑」很容易引起共鸣；再加上 Claude Code、Agent Skills 这类宿主里能直接落地的单仓库方案，收藏和转发意愿会偏高。
  • 传播：多语言 README、徽章齐、还有系统设计向的 PDF 长文，降低「这是玩具吗」的疑虑；话题本身（复刻工作风格、隐私边界）也容易被二次讨论。
  • 摩擦：星标高≠你 5 分钟能跑通——飞书/企微/钉钉接入、权限和 INSTALL 步骤，都会拉低「顺手一试」的比例；Open issues 多有时正是卡在真实集成，而不是单纯口碑差。
  • 合规：IM 数据归属公司或个人、在职/离职后的权限假设，和 README 里的「顺路径」不一定一致；星多不代表法律或人事上没有坑。

一、放到任何 Agent Skill 仓库都会问的三件事

1) Generic questions (any skill-shaped repo)

Q 我第一次要输入什么，宿主才会走这个 Skill？
A 读根目录 SKILL.md 的 frontmatter、name、触发词与 allowed-tools。本仓创建流：/create-colleague；生成后日常用带斜杠的短名调用即可（文档里常写成 /{slug}，slug = 那位同事的代号）。
Q What do I type first?
A Root SKILL.md: triggers and tools. Here /create-colleague, then day-to-day slash handles (often notated /{slug}).
Q 数据到底是 API 还是扫我电脑？
A 看 tools/：组合通常是 开放平台 REST、浏览器已登录会话（Playwright）、用户上传/导出文件；不是「偷偷全盘索引」这种主路径。
Q API or scanning my disk?
A Inspect tools/: REST + browser session + files you add — not covert full-disk indexing as the core story.
Q Work 和 Persona 为何拆开？
A README 运行顺序是 Persona 定调再 Work 执行；拆文件才能 -work / -persona 单独调用、单独迭代与纠正。
Q Why split Work vs Persona?
A Runtime Persona → Work; split enables partial invoke and cleaner corrections.

3) Scenarios the readme skims

「我都离职了，没有飞书了，还能自动生成同事 Skill 吗？」I quit Feishu — can I still auto-build a colleague skill?

核心矛盾：README 的顺路径更像「对方走、你还在」——企业 IM 仍在你的权限内。若你已离职，通常只能依赖离职前归档/导出/个人保存的副本，与「仍连租户 API 自动同步」不是同一假设。

Core: Happy path assumes tenant access. Post-exit you rely on archives/exports you hold — not live corporate sync.

「开放平台是不是就等于在网页上登录一下？」Is open-platform just “log in on the web”?

核心区分：OAuth/token 是门票；真正拉消息/文档的是后续 GET/POST。Playwright 路线是用浏览器里已有登录态读页面——两条通路，合规与稳定性权衡不同。

Core: Tokens gate REST; browser automation reuses a human session — different trade-offs.

四、真实 Issue

#65 · 人际关系名单 / 多人物路由Relationship roster / routing

open · issues/65

问题从哪来：群聊里多人插话时，希望模型知道「此刻学谁」。现有架构：一个 slug 对应一位同事的生成物；「自动认人」需要名单 + 路由策略，多半是产品层能力，而不是单改 prompt。怎么用 Issue：评估你是要「多位真人并行分身」还是「同一同事多种模式」——后者可用 -work/-persona；Issue 更贴近前者。

Origin: Multi-speaker chats need roster routing. Architecture: one slug ≈ one colleague today; true routing is a product layer. Use: Many twins vs one colleague with modes — issue leans to the former.
